# Config Manager (配置管理规范)

本 Skill 用于规范化管理项目的配置文件和环境变量，确保在本地开发、服务器部署和 GitHub 仓库之间的安全、一致和高效同步。

## 1. 本地开发环境配置阶段
- **扫描与提取**：使用 `grep` 或 AST 工具扫描整个工程（如 `process.env.*`，Python 的 `os.environ.get()`），提取所有使用到的配置项。
- **模板更新**：确保项目根目录的 `.env.example` 或对应配置文件包含所有提取出的配置项。
- **值填充**：为本地 `.env` 文件中的每个配置项填入准确的环境值，确保格式正确、无语法错误。

## 2. 服务器部署阶段
- **安全传输**：使用 `scp` 或安全的 SSH 通道将本地配置（或部署脚本）上传到服务器指定目录。
- **权限设置**：设置适当的文件权限（如 `chmod 600 .env`），确保配置信息不被越权访问。
- **验证**：在服务器上验证配置文件内容的完整性和格式正确性（例如，不应包含空密码或错误的 URL 格式）。

## 3. 版本一致性保障
- **敏感信息隔离**：严格检查 `.gitignore`，确保 `.env`、`*.key`、`*.pem` 等包含敏感信息的文件被排除在 Git 追踪之外。
- **三端同步**：
  1. 本地 `.env.example`（脱敏模板）与 GitHub 保持一致。
  2. 服务器上的实际 `.env` 与本地部署配置逻辑保持一致。

## 4. 操作流程规范
严格执行 **“本地修改 → 服务器测试 → GitHub提交”** 的三步流程：
1. 在本地完成配置修改和脚本更新。
2. 上传相关变更（脚本、非敏感配置）到服务器，并执行测试（如重启服务验证）。
3. 测试通过后，将非敏感的变更（代码、模板）提交到 GitHub 仓库。
4. **变更记录**：在提交信息或 Changelog 中记录配置修改的内容、时间和原因。

## 5. 质量控制标准
- **验证**：每次上传/应用配置前，进行缺失或错误配置项的检查。
- **回滚**：修改服务器配置前，务必先备份原配置文件（如 `cp .env .env.bak_$(date +%s)`），以便快速恢复。
- **清理**：上传测试结束后，自动清理本地和服务器产生的临时测试文件、归档日志等过程文件。

## 快速使用指令示例

**提取环境变量：**
```bash
grep -r "process.env." web/app | awk -F'process.env.' '{print $2}' | awk -F'[^a-zA-Z0-9_]' '{print $1}' | sort | uniq
```

**备份服务器配置：**
```bash
ssh user@host "cp /path/to/.env /path/to/.env.bak_$(date +%Y%m%d%H%M%S)"
```
